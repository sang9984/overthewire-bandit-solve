# Bandit Level 22 → Level 23

## Goal

A program is running automatically at regular intervals from cron, the time-based job scheduler. Look in `/etc/cron.d/` for the configuration and see what command is being executed.

**NOTE:** Looking at shell scripts written by other people is a very useful skill. The script for this level is intentionally made easy to read. If you are having problems understanding what it does, try executing it to see the debug information it prints.

## Solution

### Building on Level 21 — same pattern, more analysis

This level extends Level 21's cron pattern. The difference: the script does more than copy a file — it derives the target filename dynamically using `whoami` and `md5sum`. The challenge is reading and understanding the script, then mentally executing it as the cron would.

### Bash variable references — `$`

The `$` symbol in bash retrieves values:

|Syntax|Meaning|
|---|---|
|`$var`|Value of variable `var`|
|`${var}`|Same, with explicit boundaries|
|`$(command)`|Result of executing `command` (command substitution)|
|`$?`|Exit code of the previous command|
|`$0`, `$1`, ...|Script name, first argument, etc.|

Examples:

```bash
name="alice"
echo $name                # alice

today=$(date)             # today gets the output of `date`
echo "Today is $today"
```

### How `whoami` behaves in different contexts

`whoami` returns the **currently effective user**, which depends on **who is running the process**:

- If a user runs it manually → returns that user's name
- If cron runs a script as `bandit23` → returns `bandit23`
- If a setuid binary runs as `root` → returns `root`

This is critical for analyzing this level. The same script behaves differently depending on which user executes it.

### Pipes (`|`) and the script's hash chain

A pipe `|` sends the output of one command as the input of the next:

```bash
echo "hello" | md5sum
# 5d41402abc4b2a76b9719d911017c592  -
```

Breaking down the script's pipeline:

```bash
echo I am user $myname | md5sum | cut -d ' ' -f 1
```

Step by step:

1. `echo I am user $myname` — prints `I am user bandit23`
2. `| md5sum` — computes the MD5 hash of that string
3. `| cut -d ' ' -f 1` — `md5sum`'s output is `<hash> -`. `cut` with delimiter space and field 1 keeps only the hash.

Result: a 32-character MD5 hash string.

### Why hash the username

Hashing the username deterministically produces a filename that:

- Is unique per user
- Looks random (hard to guess for someone who doesn't know the algorithm)
- Can be regenerated by the script every time

This is a weak form of obfuscation — anyone who can read the script can compute the same hash and find the file.

### Mental execution of the script

The script:

```bash
#!/bin/bash
myname=$(whoami)
mytarget=$(echo I am user $myname | md5sum | cut -d ' ' -f 1)
echo "Copying passwordfile /etc/bandit_pass/$myname to /tmp/$mytarget"
cat /etc/bandit_pass/$myname > /tmp/$mytarget
```

When executed by cron as `bandit23`:

1. `myname` = `"bandit23"`
2. `mytarget` = MD5(`"I am user bandit23"`) = `8ca319486bfbbc3663ea0fbe81326349`
3. Reads `/etc/bandit_pass/bandit23`
4. Writes that content to `/tmp/8ca319486bfbbc3663ea0fbe81326349`

The player (bandit22) can compute the same hash manually and read the resulting file.

### Important caveat — variables don't transfer

If the player runs lines of the script manually in their own shell, the variables `myname` and `mytarget` are **not set** in that shell. Trying to use `$myname` in the player's shell will produce empty output.

To debug the script correctly:

- Replace `$myname` with the literal `bandit23` (the user the cron runs as)
- Compute the hash with that exact string
- Use the resulting hash as the target filename

## Steps

### 1. Log in as bandit22

```bash
ssh -p 2220 bandit22@bandit.labs.overthewire.org
```

Enter the password from Level 21 → 22.

### 2. Find the cron configuration for bandit23

```bash
cat /etc/cron.d/cronjob_bandit23
```

Output:

```
@reboot bandit23 /usr/bin/cronjob_bandit23.sh  &> /dev/null
* * * * * bandit23 /usr/bin/cronjob_bandit23.sh  &> /dev/null
```

Confirms: a script runs every minute as `bandit23`, located at `/usr/bin/cronjob_bandit23.sh`.

### 3. Read the cron script

```bash
cat /usr/bin/cronjob_bandit23.sh
```

Output:

```bash
#!/bin/bash
myname=$(whoami)
mytarget=$(echo I am user $myname | md5sum | cut -d ' ' -f 1)
echo "Copying passwordfile /etc/bandit_pass/$myname to /tmp/$mytarget"
cat /etc/bandit_pass/$myname > /tmp/$mytarget
```

### 4. Manually compute the target filename for bandit23

Since cron runs the script as `bandit23`, the player needs to compute the hash for that exact username. Run the same pipeline manually, substituting `bandit23` for `$myname`:

```bash
echo I am user bandit23 | md5sum | cut -d ' ' -f 1
```

Output:

```
8ca319486bfbbc3663ea0fbe81326349
```

This is the target filename in `/tmp/`.

### 5. Read the password file

```bash
cat /tmp/8ca319486bfbbc3663ea0fbe81326349
```

The output is the password for `bandit23`.

## Notes

- This level introduces **dynamic filename generation** in shell scripts. Real-world admin scripts often use hashes, timestamps, or PIDs to create unique filenames. Reading these correctly is a useful skill.
- The challenge's NOTE about "reading shell scripts written by other people" is a hint to a real skill: in penetration testing and incident response, analysts read unfamiliar scripts daily. Understanding them quickly is essential.
- The vulnerability here is the same pattern as Level 21: a sensitive file is written to a world-readable location by a higher-privileged scheduled task. Anyone who can read the script can predict the target filename and harvest the secret.
- The MD5-based filename is not security — it is at most obscurity. The script is readable, so the obscurity is trivially broken.
